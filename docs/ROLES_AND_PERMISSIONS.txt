================================================================================
РОЛИ, ПОДЧИНЕНИЯ И ПРАВА ДОСТУПА — StroyControl
================================================================================

Назначение файла: единый справочник возможностей, подчинений и прав на 
редактирование по ролям. Обновляйте этот файл при каждом запросе на изменение 
ролей или доступов.

--------------------------------------------------------------------------------
ВАЖНО: РАЗДЕЛЕНИЕ ЦЕПОЧЕК (производство и строительство)
--------------------------------------------------------------------------------

Две независимые цепочки подчинения — не путать:

  ПРОИЗВОДСТВО (модульные дома):
    Директор по производству → Мастер → Работник
    Работник отчитывается ТОЛЬКО мастеру!
    Мастер отчитывается ТОЛЬКО директору по производству!

  СТРОИТЕЛЬСТВО (каркас, газобетон, пенополистирол):
    Директор по строительству → Прораб
    Прораб отчитывается ТОЛЬКО директору по строительству!

У прораба НЕТ прямых работников в системе. Работники подчиняются ТОЛЬКО мастеру.
Прораб закрывает свои работы по этапу и работы бригад на объекте (бригады не 
являются «работниками» в системе — прораб отчитывается за выполненный объём).

1. ИЕРАРХИЯ И ПОДЧИНЕНИЯ
--------------------------------------------------------------------------------

Супер-админ (super)
    └── Администратор (admin)
            ├── Менеджер ОП (manager_op)
            ├── Директор по производству (director_production)  ← ПРОИЗВОДСТВО
            │       └── Мастер (master)
            │               └── Работник (worker)              ← только у мастера!
            └── Директор по строительству (director_construction)  ← СТРОИТЕЛЬСТВО
                    └── Прораб (foreman)                       ← работников нет

Заказчик (client) — отдельно, без подчинения

2. ВОЗМОЖНОСТИ ПО РОЛЯМ
--------------------------------------------------------------------------------

Супер-админ (super)
  - Полный доступ ко всей системе
  - Вход под любым пользователем (impersonation)
  - Создание и удаление администраторов
  - Управление всеми пользователями
  - Страница «Связи ролей» (admin/roles)

Администратор (admin)
  - Проекты: все, создание, редактирование, удаление
  - Одобрение договоров и смет (после загрузки менеджером ОП)
  - Одобрение заявок на взятие проекта (делегирование)
  - Пользователи: создание, редактирование, удаление (кроме супер-админа)
  - Этапы: все проекты
  - Заявки на редактирование: все
  - Производственный календарь: все проекты
  - Отчёты работников и прорабов
  - Справочник работ
  - Чат проектов
  - Не может: создавать других администраторов

Менеджер ОП (manager_op)
  - Проекты: только свои (созданные или назначенные ответственным)
  - Создание проектов: при создании автоматически назначается ответственным (responsible_manager_id = user_id)
  - Загрузка фото, договоров, смет (после загрузки — уведомление админу)
  - Этапы с датами начала и окончания
  - Производственный календарь: только свои проекты
  - Запрос на взятие проекта: если ответственный неактивен > 7 дней, можно запросить права (админ одобряет)
  - Не может: назначать ответственного ОП (только админ), видеть чужие проекты, одобрять документы

Директор по производству (director_production)
  - Проекты: только модульные дома (type=module) мастеров своего направления (reports_to_production_id)
  - Назначение объектов работникам (worker_project_access)
  - Заявки на редактирование: от мастеров
  - Подтверждение % выполнения работ работников
  - Подтверждение этапов модульных домов
  - Чат проекта: модульные дома своего направления
  - Справочник работ
  - Отчёты работников
  - Производственный календарь: модульные дома

Директор по строительству (director_construction)
  - Проекты: ВСЕ каркас, газобетон, пенополистирол (без ограничения по назначению)
  - Назначение любого прораба на любой объект строительства
  - Подтверждение/отклонение этапов (с комментарием при отклонении)
  - Заявки на редактирование: от прорабов
  - Чат проекта: все строительные объекты
  - Отчёты прорабов: все прорабы, все объекты
  - Производственный календарь: все строительные объекты

Мастер (master)
  - Проекты: только модульные дома (master_id = user, type = 'module')
  - Подчиняется ТОЛЬКО директору по производству (reports_to_production_id)
  - Получает и подтверждает отчёты работников (master_work_approvals)
  - Добавление отчётов по этапам
  - Заявки на редактирование этапов
  - Подтверждение этапов модульных домов (после отчёта)
  - Назначение объектов работникам (worker_project_access)
  - Чат проекта (общение с админом, менеджером ОП, директором по производству)

Прораб (foreman)
  - Проекты: назначенные директором по строительству
  - Закрытие рабочего дня: свои работы по этапу + работы бригад на объекте
  - В подчинении НЕТ работников (работники — только у мастера)
  - Отчитывается ТОЛЬКО перед директором по строительству
  - Заявки на редактирование этапов
  - Чат проекта (общение с админом, менеджером ОП, директором по строительству)

Работник (worker)
  - Проекты: только модульные, назначенные мастером или директором по производству (worker_project_access)
  - Подчиняется ТОЛЬКО мастеру (reports_to_id; никогда прорабу!)
  - Закрытие дня: работа + % + комментарий
  - Подтверждение % — мастер или директор по производству

Заказчик (client)
  - Проекты: свои (client_id = user)
  - Просмотр прогресса этапов
  - Подтверждение договора и сметы (только после одобрения админом)

РОП (rop) и Маркетолог (marketer)
  - НЕ имеют доступа к управлению проектами (admin/projects, календарь, чат проектов)
  - Доступ только к amoCRM-отчётам по назначенным правам (permission_required)
  - Декоратор admin_or_manager_op_required — только admin и manager_op для проектных маршрутов

ОТВЕТСТВЕННЫЙ МЕНЕДЖЕР ОП (отдел продаж)
--------------------------------------------------------------------------------

Поле projects.responsible_manager_id — кто ведёт проект со стороны отдела продаж.

  Создание проекта:
    • Менеджер ОП создаёт → автоматически становится ответственным (responsible_manager_id = user_id)
    • Админ или другой пользователь создаёт → выбирает ответственного из списка менеджеров ОП

  Где отображается:
    • Список проектов — колонка «Ответственный ОП»
    • Страница этапов проекта — подзаголовок «Ответственный ОП: [имя]»
    • Редактирование проекта — выбор ответственного (только админ; менеджер ОП не может менять)

  Миграция/seed: все проекты по умолчанию назначаются на Александра (менеджер ОП).
  Директор по производству: мастера и работники привязываются к bocharov.

--------------------------------------------------------------------------------
СОГЛАСОВАНИЕ ДОКУМЕНТОВ (договор, смета)
--------------------------------------------------------------------------------

Цепочка: Менеджер ОП → Админ → Заказчик

  1. Менеджер ОП загружает договор/смету в проект
     → Уведомление всем админам
  2. Админ одобряет документ (кнопка «Одобрить» на странице редактирования проекта)
     → Уведомление заказчику
  3. Заказчик подтверждает в личном кабинете
     → Документ считается согласованным

Страница «Согласование» (admin): список проектов с документами, ожидающими одобрения.

--------------------------------------------------------------------------------
ДЕЛЕГИРОВАНИЕ ПРОЕКТОВ (взятие проекта)
--------------------------------------------------------------------------------

Если ответственный менеджер ОП не проявляет активность более 7 дней (не редактирует
проект, этапы, не загружает документы), любой другой менеджер ОП может запросить
права на проект.

  1. Менеджер ОП открывает «Взять проект» — видит список доступных проектов
  2. Нажимает «Запросить» — заявка создаётся, админы получают уведомление
  3. Админ открывает «Заявки на проекты» — одобряет или отклоняет
  4. При одобрении: responsible_manager_id = заявитель, уведомления обоим менеджерам

Активность обновляется при: редактировании проекта, этапов, загрузке фото/договора/сметы.

7. ДЕЙСТВИЯ ПО РОЛЯМ (попунктно)
--------------------------------------------------------------------------------

СУПЕР-АДМИН (super)
  1. Вход под любым пользователем (impersonation)
  2. Создание и удаление администраторов
  3. Редактирование всех пользователей (роль, подчинение, телефон)
  4. Страница «Связи ролей» — просмотр иерархии
  5. Все действия администратора (проекты, этапы, заявки, отчёты, календарь, чат)

АДМИНИСТРАТОР (admin)
  1. Создание, редактирование, удаление проектов (все проекты)
  2. Назначение ответственного ОП любому менеджеру
  3. Редактирование этапов (название, плановые даты) — все проекты
  4. Одобрение/отклонение договоров и смет (после загрузки менеджером ОП)
  5. Одобрение/отклонение заявок на взятие проекта
  6. Одобрение/отклонение заявок на редактирование этапов (от мастеров и прорабов)
  7. Подтверждение этапов (модуль, каркас, газобетон, пенополистирол)
  8. Создание, редактирование, удаление пользователей (кроме супер-админа)
  9. Справочник работ: просмотр, добавление, редактирование, удаление
  10. Отчёты работников и прорабов: просмотр, удаление, редактирование закрытий дня
  11. Производственный календарь: все проекты
  12. Чат проектов: все проекты
  13. Аналитика производственного календаря

МЕНЕДЖЕР ОП (manager_op)
  1. Создание проектов (становится ответственным или создателем)
  2. Редактирование своих проектов (название, тип, адрес, заказчик, мастер, фото)
  3. Загрузка договора и сметы (уведомление админу)
  4. Удаление своих проектов
  5. Редактирование этапов своих проектов (добавление, удаление, названия)
  6. НЕ может редактировать даты этапов (только admin/superadmin)
  7. Производственный календарь: только свои проекты
  8. Чат проектов: свои проекты
  9. Запрос на взятие проекта (если ответственный неактивен > 7 дней)

ДИРЕКТОР ПО ПРОИЗВОДСТВУ (director_production)
  1. Назначение объектов мастерам и работникам своего направления
  2. Снятие назначений
  3. Одобрение/отклонение заявок на редактирование от мастеров
  4. Подтверждение % выполнения работ работников
  5. Подтверждение этапов модульных домов
  6. Справочник работ: просмотр
  7. Отчёты работников: просмотр
  8. Производственный календарь: модульные дома
  9. Чат проектов: модульные дома своего направления
  10. Аналитика цеха

ДИРЕКТОР ПО СТРОИТЕЛЬСТВУ (director_construction)
  1. ВСЕ проекты каркас/газоблок/пенополистирол (без ограничения)
  2. Назначение любого прораба на любой объект строительства
  3. Снятие назначений
  4. Подтверждение/отклонение этапов (при отклонении — комментарий, прораб дорабатывает)
  5. Отчёты прорабов: все прорабы, все объекты
  6. Производственный календарь: все строительные объекты
  7. Чат проектов: все строительные объекты
  8. Одобрение/отклонение заявок на редактирование от прорабов

МАСТЕР (master)
  1. Просмотр своих проектов (master_id = user)
  2. Добавление отчётов по этапам (фото + комментарий)
  3. Создание заявок на редактирование этапов (после выполнения)
  4. Подтверждение этапов модульных домов (после своего отчёта)
  5. Назначение объектов работникам (worker_project_access)
  6. Снятие доступа работников к объектам
  7. Подтверждение % выполнения работ работников
  8. Чат проекта: свои проекты

ПРОРАБ (foreman)
  1. Просмотр назначенных объектов (каркас/газоблок/пенополистирол)
  2. Закрытие подэтапов: минимум 3 фото, комментарий, чек-лист (если есть)
  3. Отправка этапа на проверку (когда все подэтапы закрыты)
  4. Закрытие рабочего дня (работы + % + комментарий)
  5. Создание заявок на редактирование этапов
  6. Чат проекта: назначенные объекты

РАБОТНИК (worker)
  1. Просмотр назначенных объектов
  2. Закрытие рабочего дня (работы + % + комментарий)
  3. Подтверждение этапа в каждой строке перед отправкой

ЗАКАЗЧИК (client)
  1. Просмотр своих проектов (client_id = user)
  2. Просмотр прогресса этапов и фото отчётов
  3. Подтверждение договора (только после одобрения админом)
  4. Подтверждение сметы (только после одобрения админом)

--------------------------------------------------------------------------------
8. ЗАВИСИМОСТИ (кто от кого зависит)
--------------------------------------------------------------------------------

Цепочки «блокирующих» зависимостей:

  Договор/смета:
    Менеджер ОП загружает → Админ одобряет → Заказчик подтверждает
    Риск: задержка на любом этапе блокирует следующий.

  Взятие проекта:
    Менеджер ОП запрашивает → Админ одобряет
    Риск: без админа проект не перейдёт.

  Заявка на редактирование этапа:
    Мастер/прораб создаёт → Админ или директор одобряет/отклоняет
    Риск: задержка рассмотрения — мастер не может обновить фото.

  Подтверждение этапа (модуль):
    Мастер добавляет отчёт → Мастер или директор по производству подтверждает
    Риск: без подтверждения этап не считается закрытым.

  Подтверждение этапа (каркас/газобетон/пенополистирол):
    Прораб закрывает день → Директор по строительству подтверждает
    Риск: без подтверждения этап не закрыт.

  Подтверждение % работ работника:
    Работник закрывает день → Мастер или директор по производству подтверждает
    Риск: статус «pending» до подтверждения.

  Доступ работника к проекту:
    Директор по производству или мастер назначает → Работник видит объект
    Риск: без назначения работник не может закрывать день.

  Доступ прораба к проекту:
    Директор по строительству назначает → Прораб видит объект
    Риск: без назначения прораб не видит объект.

  Создание проекта:
    Менеджер ОП создаёт → сам становится ответственным
    Админ создаёт → выбирает ответственного ОП из списка менеджеров
    Риск: некорректное назначение — не те люди получают доступ.

--------------------------------------------------------------------------------
9. РИСКИ ПО РОЛЯМ И ПРОЦЕССАМ
--------------------------------------------------------------------------------

ОБЩИЕ РИСКИ
  • Единственная точка отказа: админ — без него не одобряются документы, заявки на проекты, заявки на редактирование
  • Супер-админ: полный доступ, вход под любым пользователем — риск при утечке учётки
  • Удаление проекта: каскадное удаление этапов, отчётов, чата — необратимо

АДМИНИСТРАТОР
  • Перегрузка: много заявок (документы, редактирование, взятие проектов) — возможны задержки
  • Ошибка назначения: неправильный ответственный ОП, мастер, заказчик — некорректные права доступа
  • Удаление пользователя: каскадные последствия (проекты, назначения)

МЕНЕДЖЕР ОП
  • Потеря доступа: если админ одобрит заявку на взятие проекта — проект перейдёт другому менеджеру
  • Блокировка документов: без одобрения админа заказчик не подтвердит договор/смету
  • Не может назначить ответственного: только админ — зависимость от админа

ДИРЕКТОР ПО ПРОИЗВОДСТВУ / ПО СТРОИТЕЛЬСТВУ
  • Неверная привязка: мастер/прораб с reports_to_* = NULL не попадёт в выборку
  • Конфликт типов: директор по производству не видит каркасные проекты, по строительству — модульные

МАСТЕР / ПРОРАБ
  • Задержка заявок: админ/директор не рассмотрел — нельзя обновить фото этапа
  • Нет назначенных работников/объектов: мастер не может назначить, если директор не создал связь

РАБОТНИК
  • Зависимость от мастера: без назначения на объект — нет доступа
  • Подтверждение %: мастер или директор могут задержать — статус pending

ЗАКАЗЧИК
  • Блокировка: не может подтвердить договор/смету до одобрения админом
  • Только просмотр: не может влиять на этапы, чат, отчёты

--------------------------------------------------------------------------------
10. ВОЗМОЖНЫЕ КОНФЛИКТЫ
--------------------------------------------------------------------------------

КОНФЛИКТ 1: Два менеджера ОП запрашивают один проект
  Ситуация: проект без активного ответственного > 7 дней, оба нажали «Запросить»
  Поведение: создаются две заявки, админ одобряет одну — вторая остаётся pending
  Рекомендация: админ рассматривает по очереди; при одобрении первой — вторую отклонить

КОНФЛИКТ 2: Мастер и директор по производству — кто подтверждает этап
  Ситуация: оба могут подтвердить этап модульного дома
  Поведение: первый нажавший подтверждает, второй видит «уже подтверждено»
  Риск: нет — идемпотентно

КОНФЛИКТ 3: Админ и директор — заявка на редактирование
  Ситуация: заявка от мастера — видят админ и директор по производству
  Поведение: первый одобривший/отклонивший решает; второй получит «уже обработано»
  Риск: низкий — обычно один рассматривает

КОНФЛИКТ 4: Менеджер ОП редактирует проект, пока админ одобряет взятие
  Ситуация: менеджер А неактивен 7 дней, менеджер Б запросил, админ одобрил; менеджер А потом заходит
  Поведение: проект уже у Б, А теряет доступ (уведомление о передаче)
  Риск: менеджер А может не заметить уведомление — потеря контекста

КОНФЛИКТ 5: Работник привязан к мастеру и к директору по производству
  Ситуация: reports_to_id = мастер, reports_to_production_id = директор
  Поведение: работник отчитывается мастеру; директор подтверждает % и назначает объекты
  Риск: если reports_to_production_id не совпадает с мастером работника — фильтры могут дать пустой список

КОНФЛИКТ 6: Проект без мастера или без заказчика
  Ситуация: при создании не выбран мастер/заказчик (валидация не пропустит)
  Поведение: форма требует все поля
  Риск: при миграции/импорте возможны NULL — нужна проверка

КОНФЛИКТ 7: Чат — много участников, разные роли
  Ситуация: в чате проекта — админ, менеджер ОП, мастер, директор, прораб
  Поведение: все пишут в общий поток, можно привязать к этапу
  Риск: информационный шум; нет «пригласить в чат» — все кто имеет доступ, видят всё
