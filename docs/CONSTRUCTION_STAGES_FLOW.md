## Предлагаемая логика (пошагово)

### 1. Создание проекта
**Кто:** Админ / Менеджер ОП

**Действия:**
- Выбирает тип: Каркас, Газоблок, Пенополистирол
- Заполняет: название, адрес, заказчик, ответственный ОП
- **Прораб (master_id):** при создании проекта поле «Мастер» для стройки = прораб. Сейчас прораб выбирается и добавляется в foreman_project_access. **Вопрос:** назначать прораба сразу при создании или только директор потом?

**Предложение:** Оставить как есть — при создании можно указать прораба (master_id), он попадёт в foreman_project_access. Директор по строительству может дополнительно назначать/снимать прорабов через свой дашборд.

**Этапы:**
- Создаются из `construction_stage_templates` по типу проекта
- Для каждого этапа создаются **подэтапы** из `construction_substage_templates` (новая таблица `project_substages`)
- При создании этапов — форма ввода **даты начала** и **даты окончания** для каждого этапа (или дефолтные значения)

### 2. Редактирование дат этапов
**Кто:** Только супер-админ (is_superadmin=1) и админ (role=admin)

**Ограничение:** Менеджер ОП не может менять даты. В `admin_project_stages` при сохранении дат — проверка роли.

### 3. Назначение прораба
**Кто:** Директор по строительству

**Уже есть:** Дашборд «Назначить объект прорабу» → foreman_project_access.

### 4. Закрытие этапа прорабом (по подэтапам)
**Кто:** Прораб (foreman)

**Логика:** Этап не закрывается полностью, пока все подэтапы и чек-листы не заполнены. Закрытие в любом порядке.

**Действия по каждому подэтапу:**
1. Прораб нажимает «Закрыть подэтап» (например, «Разбивка»)
2. Форма:
   - **Фото:** минимум 3 (обязательно)
   - **Комментарий:** обязателен
   - **Чек-лист:** если у операций подэтапа есть `quick_checklist` — показываем пункты как чекбоксы (парсим «• Пункт»). Все чекбоксы должны быть отмечены. Если операций с чек-листом нет — только фото + комментарий

**Когда все подэтапы закрыты:**
- Этап становится «Готов к проверке». Прораб нажимает «Отправить на проверку» (или этап автоматически попадает в очередь директора)
- Создаётся запись `stage_completions` (status='pending')

**Хранение:**
- `substage_completions` — закрытие каждого подэтапа (project_substage_id, foreman_id, comment, checklist_data JSON)
- `reports` — фото с привязкой к substage_completion_id (или project_substage_id)
- `stage_completions` — отправка этапа на проверку (создаётся, когда все подэтапы закрыты)

### 5. Проверка директором
**Кто:** Директор по строительству

**Действия:**
- Видит этапы со status='pending' (все подэтапы закрыты, этап отправлен на проверку)
- **Подтвердить** → confirmed_by_id, confirmed_at, status='confirmed'
- **Отклонить** → status='rejected', rejection_comment. Этап возвращается прорабу
- **Повторная отправка:** прораб дорабатывает, снова отправляет — создаётся **новая запись** в stage_completions

**Сейчас:** Только «Подтвердить». Нет отклонения.

### 6. Чат проекта
**Уже есть:** foreman_project_chat, director_project_chat — по аналогии с master_project_chat. Сообщения с привязкой к этапу (stage_id).

## Модель данных (изменения)

### Новые таблицы

```sql
-- Подэтапы в проекте (связаны с этапом проекта)
CREATE TABLE project_substages (
    id INTEGER PRIMARY KEY,
    stage_id INTEGER NOT NULL REFERENCES stages(id),
    substage_template_id INTEGER REFERENCES construction_substage_templates(id),
    name TEXT NOT NULL,
    order_num INTEGER NOT NULL
);

-- Закрытие подэтапа прорабом (каждый подэтап закрывается отдельно)
CREATE TABLE substage_completions (
    id INTEGER PRIMARY KEY,
    project_substage_id INTEGER NOT NULL REFERENCES project_substages(id),
    foreman_id INTEGER NOT NULL REFERENCES users(id),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    comment TEXT,
    checklist_data TEXT  -- JSON: { "op_1": [true,true,false], ... }
);

-- Отправка этапа на проверку директору (создаётся когда все подэтапы закрыты; при повторной отправке — новая запись)
CREATE TABLE stage_completions (
    id INTEGER PRIMARY KEY,
    stage_id INTEGER NOT NULL REFERENCES stages(id),
    foreman_id INTEGER NOT NULL REFERENCES users(id),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    comment TEXT,
    status TEXT NOT NULL DEFAULT 'pending',  -- pending | confirmed | rejected
    confirmed_by_id INTEGER REFERENCES users(id),
    confirmed_at DATETIME,
    rejection_comment TEXT
);
```

### Изменения в существующих

- `stages.planned_start_date`, `planned_end_date` — уже есть, заполнять при создании
- `reports` — добавить `substage_completion_id` (nullable) для привязки фото к закрытию подэтапа; при отсутствии — привязка к stage_id (для обратной совместимости)
- При закрытии подэтапа: создаём substage_completion, загружаем 3+ фото в reports (substage_completion_id)
- При отправке этапа на проверку: создаём stage_completion (status='pending')

## Ответы на вопросы (согласовано)

1. **Гранулярность закрытия:** Этап не закрывается полностью, пока все подэтапы и чек-листы не заполнены. Закрытие по подэтапам — каждый подэтап закрывается отдельно (фото + чек-лист если есть + комментарий). Этап готов к проверке директором только когда все подэтапы закрыты.

2. **Порядок этапов:** Разрешить закрывать в любом порядке.

3. **Назначение прораба:** При создании можно указать прораба; директор может назначать/снимать через дашборд.

4. **Редактирование дат этапов:** Только admin и superadmin. Менеджер ОП — нет.

5. **Повторная отправка после отклонения:** Новая запись в stage_completions (новая попытка).
